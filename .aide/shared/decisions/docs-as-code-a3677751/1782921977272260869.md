# docs-as-code

**Decision:** Documentation lives in the repo, is reviewed in the same PR as the change it describes, and breaks CI when broken

## Rationale

Docs that live outside the repo or ship separately from code inevitably drift. Treating docs as code — same review, same CI, same merge — keeps them honest because a broken doc fails the same gate a broken build does.

## Details

Keep documentation sources in the repository (Markdown, reStructuredText, AsciiDoc, schema files, diagram DSL). Run link-check, spell-check, and build verification in CI. Docs changes block a PR just like code changes — no 'I'll update the docs later' branch.
