# changelog-discipline

**Decision:** Maintain a user-visible CHANGELOG as the canonical record of what changed per release, grouped by release and type

## Rationale

Release notes in a blog, GitHub Releases, or a Slack channel scatter and get lost. A CHANGELOG in the repo is discoverable, diffable, and survives tool migrations.

## Details

Follow Keep a Changelog conventions (Added / Changed / Deprecated / Removed / Fixed / Security) or generate from Conventional Commits (git-cliff, release-please, release-plz, semantic-release). The entry answers 'what changed for the user?' — not 'which commits landed.' Breaking changes get their own section with a migration note.
