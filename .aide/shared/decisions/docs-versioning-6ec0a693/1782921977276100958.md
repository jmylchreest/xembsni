# docs-versioning

**Decision:** Documentation is versioned alongside the code it describes — tag, branch, or a version selector — never a single 'latest' view of a multi-version project

## Rationale

A user on version 1.x reading documentation for 2.x is one of the most common — and avoidable — sources of support load. Tying docs to the shipped version removes the guesswork.

## Details

For libraries and APIs with multiple supported versions, publish a docs site with a version selector, or serve docs from the release tag. For single-version products (internal tools, latest-only CLIs), tagging the repo is sufficient — but still avoid a 'docs/' folder that mixes versions. Landing pages should make the version the reader is viewing unambiguous.
