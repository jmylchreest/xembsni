# docs-co-location

**Decision:** Place documentation as close to what it describes as possible: symbol-level comments for APIs, package READMEs for modules, top-level docs/ only for cross-cutting topics

## Rationale

Proximity defeats rot. Documentation adjacent to its subject is far more likely to be updated when the subject changes than a central docs directory a contributor never opens.

## Details

Doc comments on exported symbols (godoc, rustdoc, typedoc, etc.) describe the API. A README next to a package explains that package's purpose and entry points. A top-level docs/ tree is reserved for concerns that genuinely span packages — architecture, release process, onboarding. Avoid mirroring the whole source tree under docs/; that is a dead-letter office.
